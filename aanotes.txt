Summary
- Purpose: Investigate why harvested catalog assets (from the platform called “rahona” / ADLs) do not match EDC/Collibra assets and fix metadata harvesting so table names, partitions, and business metadata line up.
- Key findings:
  - Over-harvesting: The current scanning/harvesting script digs into partition folders and treats partition-level paths as separate tables (e.g., delinquency_2025_05) rather than recognizing the partitioned table at the root (delinquency). This causes misaligned table names in Compass/Collibra vs EDC.
  - Partition layout: Delta transaction logs (.delta/_transaction) exist only within partition folders for some datasets, not at the root. That custom write pattern prevents reading the table from the root path.
  - Name-length issues: Long concatenated path-derived table names (including repeated container/folder components like DFS coordinates) are causing 128+ character limits and other naming problems.
  - Business metadata dependency: Business metadata is sourced from EDC. If EDC lacks business metadata for ADL assets, Collibra/Compass cannot have it either.
  - Harvesting implementation: EDC harvesting currently uses a Talend-based script (SQL + Java, packaged as a zip with jars). That harvesting logic includes a configurable pattern (partition pattern) provided by source/submitters and may handle partition detection differently.
  - Access & inventory: Some team members lack access to the curated/ADL containers needed to extract sample paths; an inventory CSV of full paths (one column) was requested for analysis.
  - Multiple problem areas: Besides ADL partitioning and naming, there are Synapse-related gaps and other metadata issues to investigate; these collectively block giving a solid completion date.
